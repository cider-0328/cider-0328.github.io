# 第1回
ソフトウェア工学は、実務から体系化された知見、知見と実践の乖離、チーム全体の知識の共有が難しいところである。  
情報学の知識がなくてもITエンジニアになれるくらいチームワークのある人が重要である。  
チームワークは自分自身でも発生する。開発が終わった後でもシステムは改良が発生する。そうなったとき、自分が作ったもの、他人が作ったもののコードを読んで理解する、させる力が重要。  
AIにソフトウェアを作らせるときにも、要件定義、バージョン管理、テストなど、監督者としての能力が求められてくる。  

# 第2回
ソフトウェアはハードウェアの上に載っていて、その中にOSがある。OSの中のBIOSはハードHDDやストレージをつないでいる。基本ソフトにはほかにもデバイスドライバやAPIが含まれており、APIがミドルウェアや応用ソフトとつながってやり取りを行っている。中でも、ミドルウェアはデータベースや通信の管理の役割をになっているため、重要である。  
ソフトウェアは、1.実行されることによって必要な特性、機能、性能を提供する命令語群、2.プログラムが適切に情報を扱うことを可能とするデータ構造、3.プログラムの操作や使用法を記述した情報の3つとして定義されている。加えて、ハードウェアと異なり劣化はしないが悪化はするという特徴がある。  
さらに、ソフトウェアは新たな環境やニーズへの適応や近代的なシステム、データベスと相互で運用できるように拡張、新しいビジネス要求の実現に向けた強化が求められる。  
ソフトウェア工学は「ソフトウェアの開発、運用、メンテナンスに対するシステマチックで規律のある、定量化できるアプローチの適用、すなわちソフトウェアに対するエンジニアリングの適用」と定義されており、ゴールとしては変化し続ける状況に対応する方法論を考え出すことだとされている。  
ソフトウェア工学は、大規模化や複雑さ、IT人材の不足、社会性と求められるミッション、不確実性の増大という観点から必要となってくる。  
Hyrumの法則というものがあり、これはあるAPIに十分な数のユーザがいるとき、APIを作った人本人が契約仕様として何を約束しているかは重要ではなく、システムの持つ観察可能な挙動に関して依存するユーザが出てくるというものである。

# 第3回
ソフトウェアライフサイクルとは、ソフトウェアを0から1に作り上げ、どのように改良し、終了させていくかの一連の流れのことである。誕生には、ニーズの発生や企画、要件定義書のまとめ、開発運用には実稼働や保守運用など、様々な分類がある。  開発の初期段階にはニーズの調査が必要不可欠である。
要件定義とは、これからやることを明確にし、やる必要のないこと(無駄なこと)を省くために必要。実現できること、できないことを明確にすることも大切である。  
次に行うのは設計であり、要件定義書から設計書へコーディングができるくらいの仕様書に落とし込む工程である。ここで、クラスやモジュール単位まで分解してWBSを作成する。  
そこから、プログラマたちが制作を行う。ここでは設計書に従ってシステムを構築していくが、自社開発・外部委託などの選択肢がある。  
外部委託の種類にはオフショア開発というものがあり、賃金の安い海外企業に外部委託を行う手法である。ただし、文化や言語、品質への意識の違いに気を付ける必要がある。  
制作が行われ、作成されたソフトはほぼバグが存在する。そのため、あらかじめテストを行い、バグを早く発見し対処することが理想である。しかし、すべてをテストすることはできないので、どの個所をどの程度テストするのかを考える必要がある。  
運用・保守の段階に入ると初めはバグなどの故障が起き、対処を行うことで減っていく。その後、安定稼働というフェーズに入っていくが、長年使い続けると摩耗による故障が起きてくる。このようなシステムの故障率を表す図をバスタブ曲線という。  
摩耗故障が起こるような段階に入った時には、制作時に担当した開発者は別のプロジェクトを行っているだろう。そんな時、故障の対処を行うのは別の人かもしれない。そんな時に必要になってくるのがトラブル対応などのドキュメンテーションだ。オフショア開発なども利用することから、文書は日本語だけでなく英語でも相手に理解できるような内容、表現で書かなくてはならない。

# 第4回
プロジェクトとは、必ず終わりのある有期性、独自の目的を達成する独自性の2つの性質を持つ仕事のことである。  
プロジェクトのような仕事に対して、アプローチの仕方は2つある。それがフォアキャスティングとバックキャスティングである。  
フォアキャスティングは、学校の勉強のような一見終わりの見えないものを順番に目標に向かって手順をこなすことで、どれだけやったかが評価される。デメリットとしては目標に到達できなかったり、遠いところに到達してしまうということが挙げられる。  
バックキャスティングは、プロジェクトのような有機性のあるものを何が必要かを計画して達成に向かうことで、何を成し遂げたかが評価される。デメリットとしては目標を明確に設定する必要があったり、目標に向けての道筋を検討する必要があるといったことが挙げられる。  
QCDという品質、コスト、納期によってソフトウェアを見える化するための手法がある。  
ソフトウェアの評価にはコードのステップ数やオブジェクト容量、システムの規模だけでなく使い勝手も大事。また、品質の管理といつリリースするかのバランスが重要。  
システムの規模の概算にはファンクションポイント法が有効であり、原価を算出することで比較を行える。  
開発のプロセスには主にウォーターフォール型、スパイラルモデル、反復型、アジャイルプロセスがある。  
ウォーターフォール型は要件定義から設計、製作、テストなどが前の工程が終わり次第進める方法で、進捗管理が容易である反面、計画に狂いが出るとのちの工程にしわ寄せがいくといった特徴がある。  
スパイラルモデルは開発を小さなフェーズに分割してフェーズごとにプロトタイプを作り、それを使ってもらった人からのフィードバックをもらって次のフェーズで改良する方法で、プロトタイプを作るに想定外の作業量がかかるかもしれないというデメリットもある。  
反復型は、ソフトウェアを機能ごとに分割し、これを反復と呼ぶ単位で管理を行う方法で、部分的に完成するのでクライアントの要求を取り入れることが簡単だったり部分的に納品も可能だったりする反面、分割に要する作業や管理が増えたり一括して稼働するものでは意味がないといった点もある。  
アジャイルプロセスは一定の期間を定め、その期間のうちに決められたタスクを進める方法で、毎日デイリースクラムで進捗を管理する。変化に対応して無駄をなくし、最適な手法で動くソフトの提供を優先しており、プロセスやツールより個人や相互作用を大事にしたり、顧客とのコラボレーションを価値としている。  
開発者にもレベルがつけられており、－1～3まであって、アジャイル開発にも一定のレベル以上でないと携われないといったことがある。  
どんな開発手法でも、メリットデメリットがあるので、その状況に合わせた手法で開発をしよう。

# 第5回
WBSとは、プロジェクトを達成するために、チームが実行する作業を要素分解し、作業順に並べた構成図のことで、これを行うことで達成することが明確になり全体の管理や計画が分かりやすくなる。  
つくり方としては、1.スコープを明確にすること2.大きな作業をグルーピングすること3.グルーピングした作業の相互の関連を考えること4.各グループの作業を洗い出すことである。  
1に関して、スコープは全体像から細部に向かって設計を行うトップダウンアプローチかつ、先述したバックキャスティングで行う。  
2に関して、データの収集や分析、可視化などを行う。  
3に関して、データの収集を行った後で分析を行う。  
4に関して、MECEの考えに則ってもれなくかぶりなくを目指すことが肝心である。
